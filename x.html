Data Science Lab

The Data Science Lab (DSlab) is a sublab containing a number of high performance machines for running complex computations and data analyses. It is available for all staff and students affiliated with the Leiden Institute of Advanced Computer Science (LIACS) and the Leiden Centre of Data Science (LCDS). The DSlab is integrated and hosted within the LIACS Research and Education Lab (REL), the specialized subnetwork of the Computer Science department of the Faculty of Science of Leiden University.

The machines within the DSlab are identified by the domain liacs.nl and maintained/managed by the DSlab committee (see below).

Requesting access

All LIACS staff and students have access to the DSlab Linux machines by default. Simply login using your ULCN account, see also Access and authentication below.

If you are not a LIACS staff member or LIACS student, you can request access by contacting the DSlab committee. This is done by sending an e-mail with the ULCN username and a few sentences about what they want to do to the DSlab committee: dslab@liacs.leidenuniv.nl. The intention is to always reply promptly and arrange access quickly.

Access and authentication

To access the machines in the DSlab, you have to be within the REL network. This can be done in various ways:

  • Work on a physical staff workstation within the REL (self-managed machine).
  • Connect your machine (from anywhere in the world) via SSH gateway ssh.liacs.nl (currently also known as silver.liacs.nl).

For more information on remote access to REL, see the documentation on general REL facilities. All authentication is done using your regular ULCN username and password.

Note that ISSC workstations and gateways are for vuw.leidenuniv.nl, which is not sufficient. One has to explicitly connect to the REL and enter the liacs.nl domain via the SSH gateway ssh.liacs.nl.

Shared storage and services

All DSlab machines share the same homedirectory, automatically mounted on all DSlab machines under

/home

This storage uses glusterfs and replicates itself on two storage-servers for high availability. Note that on ssh.liacs.nl (silver), there is a different homedirectory (shared between REL machines). However, the DSlab homedirectories are also available on ssh.liacs.nl (silver), but under /dshome, so that you can for example easily transfer files from anywhere via ssh.liacs.nl to the DSlab.

There is a special data storage server with 50 TB of storage space. This space is available on all DSlab machines through

/data

Most DSlab machines also have significant local storage on the machine itself, mounted under

/local

For all shared or local storage goes: always make a subdirectory with your username as a name, in which you work. Of course, for example to use your local workstation's IDE for development, you can mount the DSlab storage on your machine using for example sshfs.

Software

Most servers run CentOS 7. The aim is to in the future have the same set of packages installed on all servers, although this is work in progress. If you are missing a particular package or tool, send an e-mail to dslab@liacs.leidenuniv.nl and state which package and version you would like, and on which machine. To speed up the process, you are advised to do a yum search packagename yourself to see if the package can be installed effortlessly through the package manager. If it can be, we can usually install it for you in a short time. If your packacke is not available via yum, and there is no need to have the package installed for the entire system, you can of course always install it locally in your homedirectory or on the machine's local storage.

In general, CentOS does not aim to have very recent software packages. If you are confronted with this, Software Collection Libraries (SCLs) are sometimes needed to run code dependent on newer software packages. Such a library will use its version of packages if it has it, and otherwise it will default to the system's version. An example is devtoolset-8, which brings an up-to-date version of many packages, such as gcc and valgrind. To enable such a software collection manually in the bash prompt, use

scl devtoolset-8 enable bash

To exit the environment, just type exit.

Another option is to use the SCL by default. You can add this in your ~/.bashrc file:

source /opt/rh/devtoolset-4/enable

Fair use policy

For all storage, memory and CPU/GPU time goes: there is a fair use policy in place. This asks you to be reasonable with your usage, and to be open and communicative towards colleagues and students that you want to share a particular resource with.

If a user is taking up excessive amounts of resources on machines, or taking up GPUs without running experiments on them, you can email that user by finding their ULCN email. To do this, you can obtain the PID of the process taking up GPUs using nvidia-smi, and then use ps up PID. The output of that command then shows which user is running the problematic process(es). If the user is a student, you can reach them via @umail.leidenuniv.nl. If the user is staff, then run getent passwd NAME to obtain their real name, in order to look up their email address on the search box on leidenuniv.nl.
